# Spring JPA N+1 문제
- 발생 원인 : JPQL은 글로벌 페치 전략과 상관없이 SQL을 생성할 때, 연관된 엔티티를 자동으로 조인하지 않고 주 엔티티만 조회하기 때문

### Fetch Join
- 정의: 연관된 엔티티나 컬렉션을 SQL 한 번에 함께 조회(즉시 로딩) 하여 영속성 컨텍스트에 모두 담는다
  - 세밀한 쿼리 제어 가능하지만, 쿼리 문자열(JPQL)을 직접 작성해야 함
- 문제점 1: 카테시안 곱으로 조회하므로 쿼리의 결과값이 N개 발생한다
  - 특정 중복을 해결하기 위해 `DISTINCT` 키워드를 사용할 수 있다 (Hibernate 6 / Spring Boot 3 이상에서는 자동 적용)
- 문제점 2: 둘 이상의 컬렉션을 페치 조인하면 안됨 (`MultipleBagFetchException`)
  - 하나의 데이터에 N*M 개의 row로 조회되므로 데이터의 뻥튀기가 심해질 수 있으므로
  - 연관 관계 자료 구조를 Set으로 바꾸면 에러는 발생하지 않지만, 데이터의 뻥튀기가 심해지는 것을 막을 수 없다.

### `@EntityGraph`
- 정의 : `attributePaths` 속성에 함께 조회할 연관 엔티티의 필드명을 적어주는 것
- 구현 : 기본적으로 외부 조인(Left Outer Join)을 사용
  - 연관된 데이터가 없는 주 데이터는 조회 결과에서 아예 누락될 위험이 있음
- 장점 : 코드가 간결함, JPQL 작성 불필요
- 단점 : 복잡한 쿼리나 최적화에는 한계가 있음

### Batch Size 설정
- 정의 : 지연 로딩(Lazy Loading)을 하되 Batch Size 옵션으로 성능을 최적화하는 방법
- 장점 : 컬렉션 조회 시 페이징 처리가 필요할 때 매우 유용
- 단점 : N+1을 완전히 없애진 않고 최소화함
