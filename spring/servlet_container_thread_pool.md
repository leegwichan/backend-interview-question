# 스레드 풀

### 역할 및 구조
- 미리 생성된 스레드를 재사용함으로써 스레드 컨텍스트 스위칭에 드는 오버헤드를 줄여 애플리케이션의 성능을 향상
- 작동 방식
  1. 클라이언트로부터 HTTP 요청이 들어오면, 서블릿 컨테이너는 이 요청을 처리할 준비가 된 스레드를 풀에서 가져온다.
  2. 요청 처리 스레드는 해당 HTTP 요청을 받고, 이를 Spring MVC의 DispatcherServlet에 전달하고 응답을 받는다.
  3. 요청 처리가 완료되면, 스레드는 클라이언트에게 HTTP 응답을 반환하고, 곧바로 스레드 풀로 반납됩니다.

### 스레드 풀 설정
- `minSpareThreads`(core) : 스레드 풀이 평소에 최소한으로 유지하는 스레드의 수
 - 요청이 들어오는 즉시 처리할 수 있어 시스템의 응답 속도를 일정하게 유지하는 데 도움이 된다.
- `maxThreads`(max): 스레드 풀이 가질 수 있는 최대 스레드 수
 - 이 값이 곧 서버가 동시에 처리할 수 있는 최대 요청 수와 직결
- queue : core 만큼의 스레드가 모두 사용 중일 때, 새로 들어온 작업 요청들을 일시적으로 저장하여 대기시키는 공간
 - 큐의 크기에 제한을 둘 수도 안둘 수도 있다. (제한이 있으면 처리량 감소, 제한이 없으면 타임아웃 증가)

### 스레드 포화 정책 (`@Async`)
- AbortPolicy (기본 정책)
  - 동작: 스레드 풀이 포화 상태일 때 새로운 작업을 거부하고 `RejectedExecutionException` 예외를 발생시킨다.
  - 사용: 예외를 통해 명확하게 스레드 풀 한계 도달을 알리고 싶을 때 사용
- CallerRunsPolicy (호출자 실행 정책)
  - 동작: 거부된 작업을 호출 스레드가 직접 실행합니다.
  - 사용: 비동기 작업 처리 속도가 느려지더라도 요청을 잃지 않고 처리해야 할 때 (단, 호출 스레드에 부담을 줄 수 있다)
- DiscardPolicy (작업 버리기 정책)
  - 동작: 새로운 작업을 그냥 무시하고 버린다.
  - 사용: 중요하지 않은 작업이 밀릴 경우, 과부하를 막기 위해 단순히 버려도 되는 작업
- DiscardOldestPolicy (가장 오래된 작업 버리기 정책)
  - 동작: 대기 큐에서 가장 오래된 작업을 버리고, 그 자리에 새로운 작업을 넣는다.
  - 사용: 실시간성이 중요한 작업에서 더 최신 작업을 처리하는 것이 중요할 때
