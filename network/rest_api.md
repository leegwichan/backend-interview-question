# REST API

### REST API
- REST (Representational State Transfer API)
  - HTTP URI(Uniform Resource Identifier)를 통해 자원(Resource)을 명시한다
  - HTTP Method를 통해 해당 자원(URI)에 대한 CRUD 기능을 적용한다.

- [REST의 특징](https://aws.amazon.com/ko/what-is/restful-api/)
  - Server-Client(서버-클라이언트 구조)
  - Stateless(무상태)
  - Cacheable(캐시 처리 가능)
  - Layered System(계층화)
  - Uniform Interface(인터페이스 일관성)

- REST API : REST 아키텍처의 제약 조건을 준수하는 애플리케이션 프로그래밍 인터페이스

### HTTP Method
- HTTP Method
  - 해당 자원(URI)에 대한 CRUD 기능을 의미한다
  - Method를 분리하는 이유 : 리소스(URI)와 동작(Method)을 분리하기 위해

- 종류
  - GET : 해당 자원을 조회
  - POST : 해당 자원을 등록(추가)
  - PUT : 리소스를 완전히 대체 (해당 리소스가 없다면, 새롭게 생성)
  - PATCH : 리소스를 부분적으로 변경함
  - DELETE : 해당 자원을 삭제
  - 이외, HEAD, OPTIONS, CONNECT 등이 있다

- GET vs POST
  - GET
    - 리소스를 조회하는 메서드
    - 서버에게 데이터를 전달하는 경우, 쿼리 스트링을 통해 전달함
    - 성공적으로 조회시, 200 OK 응답를 받음
    - ex) `GET /members?sort=age`
  - POST
    - 새로운 리소스를 생성하는데 사용
    - 서버에게 데이터를 전달할 때, Body에 정보를 담아 전송한다
    - 성공적으로 생성시, 201 CREATED 응답을 받음

- PUT vs PATCH
  - PUT
    - 리소스를 완전히 덮어 씌운다
    - 멱등성을 지닌다
  - PATCH
    - 리소스를 부분 변경하는데 사용한다
    - 멱등성을 지니지 않는다
  - 멱등성 : 같은 요청을 반복했을 때, 매번 같은 결과가 나오는 특징

### 비멱등성 API 중복 호출 문제

#### 멱등성 키 (Idempotency Key) 패턴
- 개념: 클라이언트가 요청을 보낼 때, 헤더에 고유한 키(ex. UUID)를 담아 보냅니다.

- 구현 방법
  1. 요청 수신 및 키 확인
  2. Redis Atomic Lock 시도
  3. 비즈니스 로직 수행
  4. Redis에 결과 업데이트 / 비즈니스 로직 실패 시 예외 처리

- 엣지 케이스 고려하기
    | 상황 | 문제점 | 해결 방안 |
    | :--- | :--- | :--- |
    | 서버 다운 | 로직 수행 중 서버가 죽어서 Redis 키가 `PROCESSING` 상태로 영원히 남음. | TTL(만료 시간) 설정 필수. 1분\~5분 뒤에 키가 자동 삭제되게 하여 재시도 가능하게 함. |
    | TTL 만료 | 로직이 너무 오래 걸려서 처리가 끝나기도 전에 Redis 키가 만료되어 락이 풀림. | 로직의 타임아웃 시간을 정확히 계산하여 TTL을 넉넉히 잡거나, 별도의 갱신 로직 필요. |
    | 결과 캐싱 | 중복 요청 시, 매번 DB를 조회하면 부하가 생김. | 처리가 완료되면 Redis에 응답 데이터 자체(JSON)를 저장해두고, 중복 요청 시 DB 조회 없이 Redis 값만 반환. |

- 키 갱신 시점
    | 상황 | 키 유지 여부 | 설명 |
    | :--- | :--- | :--- |
    | 네트워크 오류 / 타임아웃 | 유지 | 요청이 서버에 닿았는지 모르므로, 동일한 키로 재시도해야 멱등성이 동작함. |
    | 4xx 에러 | 갱신 (보통) | "잔액 부족" 등으로 실패했다면, 유저가 금액을 바꾸고 다시 시도해야 하므로 새로운 요청(새 키)이어야 함. |
    | 명시적 수정 (뒤로가기) | 갱신 | 유저가 [이전 단계]로 돌아가서 금액이나 받는 사람을 수정했다면, 이건 새로운 의도이므로 새 키 생성. |
    | 요청 성공 | 삭제/갱신 | 송금이 완료되어 '완료 페이지'로 갔다가, 다시 '송금 페이지'로 돌아오면 새 키 생성. |

#### 데이터베이스 관리 포인트
- DB Unique Index
  - 특정 컬럼에 유니크 인덱스를 걸어두어 중복 방지
  - MySQL 기준 유니크 인덱스를 걸어두면, 해당 컬럼의 세컨더리 인덱스를 자동으로 만들어두어 B+ Tree를 통해 중복 검사를 한다.
- `INSERT IGNORE` 전략 : PK 등이 중복되었을 경우, 해당 INSERT 무시 (Spring Data JPA에서는 안될 수 있음)
- `@Version`을 이용한 낙관적 락, 비관적 락 우선 조회 등을 이용하여 동시 상황 관리
