# 인증과 인가

### 인증의 세션과 토큰 방식
| 구분 | Session (세션) | Token (토큰/JWT) |
| --- | --- | --- |
| 저장 위치 | 서버 메모리/DB | 클라이언트 (저장소) |
| 상태 유지 | Stateful (상태 유지) | Stateless (무상태) |
| 확장성 | 낮음 (서버 분산 시 복잡) | 높음 (확장성 유리) |
| 인증 방식 | Session ID를 쿠키에 담아 통신 | Access Token을 헤더에 담아 통신 |

### JWT (JSON Web Token)
- 정보를 안전하게 전송하기 위해 정의된 개방형 표준, Header.Payload.Signature 세 부분을 점으로 구분한 문자열
- Header : typ - 토큰 타입(JWT), alg - 시그니처 해싱을 위한 알고리즘(HS256, RS256)
- Payload : 토큰에 담을 정보
  - Registered Claims: 토큰 정보를 위해 이미 정해진 이름들 (iss: 발행자, exp: 만료시간, sub: 제목 등)
  - Public/Private Claims: 사용자가 정의하는 정보 (ex. userId, role)
  - 비밀번호 같은 민감한 정보는 절대 넣으면 안된다.
- Signature : 토큰의 무결성을 확인
  - 생성 방식: 인코딩된 Header + 인코딩된 Payload + 서버가 가진 비밀 키를 합친 후, 헤더에서 지정한 알고리즘으로 해싱
  - 역할: 서버는 전달받은 토큰의 Header와 Payload를 자신의 비밀 키로 다시 계산해보고, 이 시그니처 값과 일치하는지 확인

### OAuth
- 사용자가 자신의 비밀번호를 제3자 앱에 공유하지 않고, 자신이 소유한 다른 서비스의 데이터에 접근할 수 있는 권한을 부여하기 위한 개방형 표준 프로토콜

- 4가지 핵심 역할
  - 사용자 : 리소스의 주인 (예: 구글 계정 소유자)
  - 클라이언트 : 사용자의 리소스를 사용하고자 하는 제3자 애플리케이션
  - 데이터 서버 : 사용자의 데이터를 가지고 있는 서버 (예: 구글 API 서버)
  - 인증 서버 : 권한을 부여하고 액세스 토큰을 발급해주는 서버 (예: 구글 인증 서버)

- 주요 동작 흐름 (Authorization Code Grant 기준)
  1. 권한 요청: 사용자가 우리 서비스에서 '구글 로그인' 버튼을 클릭하면, 브라우저가 구글 인증 서버의 로그인 페이지로 이동
  2. 로그인 및 동의: 사용자가 구글에 로그인하고, 우리 서비스가 요청한 정보(이메일, 프로필)에 대한 접근 권한을 승인
  3. Authorization Code 발급: 구글 서버는 우리 서비스의 Redirect URI로 권한 부여 코드(Authorization Code)를 보냄
  4. 액세스 토큰 교환: 우리 서버는 받은 코드를 가지고 구글 인증 서버에 직접 요청을 보내 액세스 토큰을 받음, 이 때 Client Secret을 함께 보내 신원을 확인
  5. 리소스 접근: 이제 우리 서버는 이 Access Token을 사용해 구글 리소스 서버에서 필요한 사용자 정보를 가져옴
