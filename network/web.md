# Web

### 도메인에 접속하면 일어나는 일

#### 1. DNS Lookup: 도메인 네임을 IP 주소로 변환
1.  캐시 확인 (Hierarchy)
    1. Browser Cache: 이전에 방문한 기록이 있는지 확인합니다.
    2. OS Cache: `hosts` 파일이나 시스템 콜을 통해 OS 수준의 캐시를 확인합니다.
    3. Router/ISP Cache: 공유기(Router)와 인터넷 서비스 제공자(ISP)의 DNS 서버 캐시를 확인합니다.
2.  Recursive Query (재귀적 쿼리): 캐시에 없다면 ISP의 DNS 서버가 '뿌리'부터 차례대로 물어봅니다.
    1.  Root DNS 서버: ".com"을 담당하는 TLD 서버 주소를 알려줍니다.
    2.  TLD(Top-Level Domain) 서버: "naver.com"을 담당하는 권한 있는(Authoritative) DNS 서버 주소를 알려줍니다.
    3.  Authoritative DNS 서버: 최종적으로 "[의심스러운 링크 삭제됨]"의 실제 IP 주소를 반환합니다.

#### 2. TCP 연결 및 HTTPS 핸드셰이크
1. TCP 3-Way Handshake: 전송 계층에서 클라이언트와 서버 사이의 논리적인 연결을 확립합니다. (SYN -> SYN/ACK -> ACK)
2. TLS/SSL Handshake: 네이버는 HTTPS를 사용하므로, 데이터를 암호화하기 위한 대칭 키를 안전하게 공유하는 과정이 추가됩니다. 이 과정을 통해 보안 연결이 설정됩니다.

#### 3. HTTP Request 전송
- 메시지 생성: 브라우저가 `GET / HTTP/1.1`과 같은 HTTP 요청 메시지를 생성합니다.
- 캡슐화: 메시지는 프로토콜 계층을 내려가며 패킷으로 쪼개집니다.
  - Transport 계층: 포트 번호 추가 (TCP)
  - Network 계층: IP 주소 추가 (IP)
  - Link 계층: MAC 주소 추가 (Ethernet)
- 라우팅: 라우터를 거치며 Next Hop을 찾아 네이버 서버가 있는 곳까지 이동합니다.

#### 4. 서버 처리 및 HTTP Response 전송
- 서버 로직 수행: 서버는 요청을 분석하고 DB 조회 등이 필요하면 수행한 후, HTML 문서나 JSON 데이터를 생성합니다.
- 응답 메시지 전송: 생성된 데이터를 HTTP Response 메시지에 담아 다시 클라이언트에게 보냅니다. 과정은 요청 시와 마찬가지로 패킷 캡슐화 및 라우팅을 거칩니다.

#### 5. 브라우저 렌더링
1. Parsing: HTML을 읽어 DOM 트리를 만들고, CSS를 읽어 CSSOM 트리를 만듭니다.
2. Render Tree 생성: DOM과 CSSOM을 결합하여 화면에 실제로 표시될 요소들로만 구성된 렌더 트리를 만듭니다.
3. Layout (Reflow): 각 노드가 화면의 어디에, 어떤 크기로 배치될지 계산합니다.
4. Paint (Repaint): 계산된 위치에 실제 픽셀을 채워넣고 이미지를 그립니다.

### DNS 라운드 로빈 방식
- 별도의 소프트웨어 혹은 하드웨어 로드밸런싱 장비를 사용하지 않고, DNS만을 이용하여 도메인 레코드 정보를 조회하는 시점에서 트래픽을 분산하는 기법
   - 도메인을 사용하는 모든 서비스에서 사용이 가능 (FTP, SMTP, TURN 등)

- 동작 원리
  - 클라이언트 요청: 사용자가 도메인에 접속하기 위해 DNS 쿼리를 보낸다.
  - DNS 서버는 등록된 IP 주소 리스트를 가지고 있습니다. 요청이 올 때마다 이 리스트의 순서를 순환시키며 응답
  - 대부분의 브라우저는 리스트의 가장 첫 번째 IP를 사용하여 접속을 시도, 결과적으로 유저들은 서로 다른 서버로 접속하게 되어 부하가 분산

- 장점
  - 별도의 로드 밸런싱 장비를 구축할 비용이나 인력이 없어도 DNS 설정만으로 구현 가능
  - 비용 효율적: 추가적인 인프라 비용이 거의 들지 않음
- 단점
  - 서버 상태 확인 불가 (No Health Check) : 특정 서버가 다운되어도 DNS는 해당 서버의 IP를 계속 알려주기 때문에, 일부 유저들은 접속 실패를 경험
  - DNS 캐싱 문제 : 브라우저나 ISP(인터넷 서비스 제공자)의 캐시에 저장, TTL 시간 동안은 똑같은 IP로만 접속하기 때문에, 부하 분산이 실시간으로 균등하게 이루어지지 않을 수 있다.
  - 가중치 부여 어려움 : 서버마다 성능이 다를 수 있는데, 무조건 1:1로 분산하기 때문에 효율적인 자원 사용이 어려움

### CORS
- 추가 HTTP 헤더를 사용하여, 한 출처에서 실행 중인 웹 애플리케이션이 다른 출처(Origin)의 자원에 접근할 수 있는 권한을 부여하도록 브라우저에 알려주는 메커니즘
- 배경: SOP (Same-Origin Policy)
  - 동일 출처 정책. 브라우저가 보안을 위해 동일한 출처의 리소스만 상호작용하도록 제한하는 기본 보안 원칙
  - CSRF(Cross-Site Request Forgery)나 XSS 같은 보안 공격을 방지하여 사용자의 정보를 보호
  - 현대 웹은 여러 도메인 간의 데이터 통신이 필수적이므로, CORS를 도입함
- Origin 판단 기준 : Protocol(`https`), Host(`naver.com`), Port(`443`)
- CORS 동작 방식 : Preflight Request(예비 요청)
  - 라우저는 실제 요청을 보내기 전, 안전한지 확인하기 위해 OPTIONS 메서드로 예비 요청을 먼저 보낸다. (Origin, Access-Control-Request-Method 포함)
- 주의사항 : 보안을 위해 Allow-Origin: *은 피하고, 필요한 도메인만 화이트리스트로 관리해야 한다.

### 브라우저 저장소 종류
| 구분 | 쿠키 | 로컬 스토리지 | 세션 스토리지 |
| --- | --- | --- | --- |
| 데이터 유지 | 만료 날짜 전까지 유지 | 영구적으로 유지 | 탭/창을 닫으면 삭제 |
| 용량 | 매우 작음 (약 4KB) | 큼 (약 5MB) | 큼 (약 5MB) |
| 서버 전송 | 모든 HTTP 요청 시 자동 전송 | 전송되지 않음 (클라이언트 전용) | 전송되지 않음 (클라이언트 전용) |
| 주요 용도 | 세션 ID 저장, 팝업 '오늘 그만보기' | 자동 로그인, 사용자 설정 저장 | 비로그인 장바구니, 일시적인 데이터 |

- 특징
  -  쿠키는 서버와 통신할 때마다 헤더에 담겨 가기 때문에, 너무 많은 데이터를 넣으면 네트워크 성능이 저하될 수 있음을 언급하세요.
  - Web Storage는 HTML5부터 등장했으며, 쿠키의 적은 용량과 불필요한 서버 전송 문제를 해결하기 위해 사용합니다
