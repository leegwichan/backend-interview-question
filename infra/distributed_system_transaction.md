# 분산 환경에서의 트랜잭션

### 트랜잭션 아웃박스 패턴
- 목적 : 데이터베이스에 데이터를 저장하는 작업과 메시지 큐에 이벤트를 발행하는 작업을 원자적으로 처리하기 위한 기법

- 단계별 흐름
  1. 트랜잭션 시작: 로컬 DB 트랜잭션을 시작
  2. 비즈니스 데이터 저장: Orders 테이블에 주문 정보를 INSERT 합니다.
  3. 아웃박스 데이터 저장: 같은 트랜잭션 안에서, 보낼 메시지 내용을 Outbox라는 별도의 테이블에 INSERT
  4. 트랜잭션 커밋: 이 두 작업은 같은 DB 트랜잭션이므로, 둘 다 성공하거나 둘 다 실패한다.
  5. 메시지 릴레이: 별도의 프로세스가 Outbox 테이블을 감시하다가, 새로운 메시지가 보이면 이를 읽어서 메시지 브로커로 전송
  6. 완료 처리: 전송이 성공하면 Outbox 테이블에서 해당 데이터를 삭제하거나 '전송 완료' 상태로 변경

- 메시지 릴레이 구현 방식 
  - 폴링 발행기 : 주기적으로 SELECT 쿼리를 날려 데이터를 읽음 (DB 부하, 실시간성이 떨어짐)
  - 트랜잭션 로그 테일링 : DB의 트랜잭션 로그를 실시간으로 읽어 변경 사항 감지 (설정과 운영 복잡도가 높음, 특정 DB 기술에 종속)

- 핵심 고려 사항
  - 적어도 한 번 전송 (At-least-once Delivery) : 메시지 릴레이 전에 프로세스가 죽어 재전송 가능성 있음, 컨슈머는 멱등성을 보장하도록 설계
  - 메시지 순서 보장 : 엄격한 순서가 필요한 경우, 릴레이 프로세스가 단일 스레드로 동작하거나, 파티셔닝 전략을 잘 세워야 한다.

### Saga 패턴
- 목적 : 분산된 환경에서 긴 트랜잭션의 데이터 일관성을 관리하기 위한 패턴
  
- 단계별 흐름
  1. 각 서비스는 자신의 DB에서 로컬 트랜잭션을 실행하고, 다음 서비스 트리거를 위한 이벤트를 발행
  2. 만약 과정 중간에 에러가 발생하면, 앞서 완료된 트랜잭션들을 취소하기 위한 반대 작업(보상 로직)을 역순으로 실행

- 코레오그래피(Choreography) - 이벤트 기반 구현 방식
  - 동작: 서비스 A가 "주문 생성됨" 이벤트를 던지면, 서비스 B가 이를 구독하여 "결제 처리"를 하고, 결과 이벤트를 다시 던진다
  - 장점: 구성이 간단하고 서비스 간 결합도가 낮다
  - 단점: 프로세스가 복잡해지면 누가 어떤 이벤트를 기다리는지 파악하기 어려워진다(순환 의존성 위험)

- 오케스트레이션 (Orchestration) - 지휘자 기반 구현 방식
  - 동작: 오케스트레이터가 서비스 A에게 "주문해"라고 명령(Command)을 보낸다. A가 "완료" 응답을 주면, 오케스트레이터가 서비스 B에게 "결제해"라고 명령
  - 장점: 비즈니스 흐름을 한곳에서 파악하고 관리하기 쉽다. 복잡한 로직에 적합
  - 단점: 오케스트레이터에 로직이 집중될 수 있고, 추가적인 인프라 구축이 필요할 수 있다

- 한계 (ACID 비보장)
  - 중간 상태 노출: 전체 프로세스가 끝나기 전에도, 다른 사용자는 중간 상태의 데이터를 볼 수 있다
  - 데이터 정합성: SAGA는 즉각적인 일관성보다는 결과적 일관성(Eventual Consistency)을 목표로 한다
